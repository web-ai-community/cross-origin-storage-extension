Cross-Origin Storage — Test Page

This page tests navigator.crossOriginStorage.requestFileHandle() (singular, current API) and the deprecated plural navigator.crossOriginStorage.requestFileHandles() (see #61). It requires the Cross-Origin Storage extension to be installed and enabled.

Known issue under test: when requesting multiple resources and only some exist, the non-existent handles return a File whose content is the string "undefined" instead of throwing a NotFoundError.

Singular requestFileHandle() tests (current API) appear first; the deprecated plural requestFileHandles() tests follow at the bottom of this table.

# Scenario Status Run Detail

Worker Initialization Variants

Each row creates a worker using a different URL scheme or type and runs a minimal store-then-read COS operation to confirm that navigator.crossOriginStorage is correctly injected in every case.

  • blob: URL, classic & module — tested twice each: once with the URL revoked before the worker starts (regression for the extension's sync-XHR inline path) and once with it kept live (the normal path).
  • data: URL, classic only — the module variant is not covered because browsers reject data: URLs as module-worker sources (opaque origin).
  • Same-origin HTTP, classic & module — the module variant imports a helper via a relative path, which is a regression test for the bug where inlining the worker source changed the base URL and broke import resolution.
  • Cross-origin HTTP, classic & module — both served from cdn.jsdelivr.net (cross-origin to the GitHub Pages host, application/javascript MIME type, CORS Access-Control-Allow-Origin: *).
  • SharedWorker, classic & module — the extension patches window.SharedWorker with the same wrapper-blob technique, adapted to the onconnect / MessagePort API. A thin adapter in create() presents the same interface to the test harness.
  • Nested worker — spawns a second worker from inside the first to verify that injection propagates to dynamically created sub-workers.
  • Not covered — ServiceWorker: Chrome rejects blob: URLs as service-worker script sources, so the wrapper-blob technique does not apply. Patching navigator.serviceWorker.register() is possible but requires a different relay mechanism (service workers communicate via Clients.matchAll() + client.postMessage(), not a direct MessageChannel), and is left for a future implementation.

Stress Tests

Large-file tests that exercise the streaming-hash write path and the Blob-based read path. Each test creates a file on the fly, stores it in COS, reads it back, then deletes it from COS (and therefore from local disk) before reporting a result. They take several minutes on a typical machine.

# Scenario Status Run Detail

Origins Visibility & PHL Gate Tests

Tests for the three origins visibility tiers — '*' (global), an explicit list, and omitted (same-site) — and the Public Hash List gate that applies to global resources.

The last two tests load the extension relay page in a hidden iframe. Because it is at a chrome-extension:// URL it is always a different origin from any web page, acting as a genuinely cross-origin reader.

# Scenario Status Run Detail

Multi-Origin PHL Gate Tests

Tests the full cross-origin read matrix for all three origins tiers across a same-origin, a same-site subdomain, and a cross-site origin. Sub-origin reader pages are loaded in hidden iframes using the same postMessage relay pattern as the tests above.

These tests require the a.test / sub.a.test / b.test multi-origin environment with a pre-seeded mock Public Hash List — the setup provided automatically by e2e-test.mjs. They are n/a on GitHub Pages and other single-origin hosts.

⚠️ Multi-origin setup not detected (location.hostname is not a.test). These tests run automatically via e2e-test.mjs but show n/a here.

CSS cross-origin-storage() Polyfill Tests

Tests the CSS cross-origin-storage() modifier polyfill. A <link rel="stylesheet" data-cos> in this page's <head> points to test-cos-fonts.css, which uses the proposed url("…" integrity("sha256-…") cross-origin-storage(*)) syntax. The extension intercepts the link, serves the font bytes from COS (fetching from the network and caching on first run), creates a page-origin blob URL, and re-injects the rewritten rule so the browser can load it normally.

Declarative HTML Integration Tests

Tests the <link integrity crossoriginstorage> / <script integrity crossoriginstorage> polyfill (see the README's "Declarative HTML Integration" section). The static <link> and <script> in this page's <head> point to test-declarative-style.css and test-declarative-script.js with a bare (same-site-only) crossoriginstorage attribute. The last two tests are cross-origin: they run inside the sub.a.test / b.test iframes already used by the multi-origin PHL tests above, and deliberately point a <link> at a non-existent URL so a pass is only possible if the stylesheet was actually served from COS rather than falling back to a native fetch (they use <link>, not <script>, for this specific check: a classic script's src can't be overridden after the browser starts fetching it, so pointing one at a broken URL would just fail regardless of whether COS sharing works — see the comment on processDeclarativeResource in main-world.js for the full explanation).

Declarative JavaScript Integration Tests

Tests import attributes (see the README's "Declarative JavaScript Integration" section): import x from "url" with { integrity, crossOriginStorage } and import(url, { with: { integrity, crossOriginStorage } }). Real browsers reject any unrecognized import-attribute key with a synchronous TypeError before any fetch is even dispatched, and dynamic import() can't be monkey-patched the way fetch/Worker can — so, following the approach pioneered by es-module-shims, this polyfill only activates for <script type="module-cos"> (a type real browsers never try to parse), rewriting its source's import specifiers before executing it as a real type="module" script. A navigator.crossOriginStorage.__non_standard__import() helper covers dynamic, runtime-computed specifiers instead. The last two tests are cross-origin, reusing the sub.a.test/b.test iframes above, and call the helper with a non-existent URL so a pass is only possible via COS.
