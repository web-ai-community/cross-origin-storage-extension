Cross-Origin Storage — Test Page

This page tests navigator.crossOriginStorage.requestFileHandles() for the five combinations of existing and non-existing resources. It requires the Cross-Origin Storage extension to be installed and enabled.

Known issue under test: when requesting multiple resources and only some exist, the non-existent handles return a File whose content is the string "undefined" instead of throwing a NotFoundError.

# Scenario Status Run Detail

Worker Initialization Variants

Each row creates a worker using a different URL scheme or type and runs a minimal store-then-read COS operation to confirm that navigator.crossOriginStorage is correctly injected in every case.

  • blob: URL, classic & module — tested twice each: once with the URL revoked before the worker starts (regression for the extension's sync-XHR inline path) and once with it kept live (the normal path).
  • data: URL, classic only — the module variant is not covered because browsers reject data: URLs as module-worker sources (opaque origin).
  • Same-origin HTTP, classic & module — the module variant imports a helper via a relative path, which is a regression test for the bug where inlining the worker source changed the base URL and broke import resolution.
  • Cross-origin HTTP, classic & module — both served from cdn.jsdelivr.net (cross-origin to the GitHub Pages host, application/javascript MIME type, CORS Access-Control-Allow-Origin: *).
  • SharedWorker, classic & module — the extension patches window.SharedWorker with the same wrapper-blob technique, adapted to the onconnect / MessagePort API. A thin adapter in create() presents the same interface to the test harness.
  • Nested worker — spawns a second worker from inside the first to verify that injection propagates to dynamically created sub-workers.
  • Not covered — ServiceWorker: Chrome rejects blob: URLs as service-worker script sources, so the wrapper-blob technique does not apply. Patching navigator.serviceWorker.register() is possible but requires a different relay mechanism (service workers communicate via Clients.matchAll() + client.postMessage(), not a direct MessageChannel), and is left for a future implementation.
